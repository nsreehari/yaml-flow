Assumptions and Tradeoffs (Authoritative Notes)

  • Runtime artifact contract is intentionally minimal: schema_version, card_id, and computed_values. Rich context like fetched_sources is transport/runtime detail, not persistence contract.
  • Card definitions are treated as source-of-truth for authored state (card_data). Runtime artifacts capture evaluated outputs, not authored configuration.
  • Source definitions are intentionally opaque to the framework. Executor-specific fields are owned by domain teams and passed through without interpretation.
  • Token-level interoperability is the stable boundary: producers emit via provides.bindTo, consumers depend via requires. Domain logic should evolve behind this boundary.
  • Server and browser shells are equivalent by intent, not byte-identical implementations. Browser mode prioritizes local iteration; server mode prioritizes process boundaries and operational realism.
  • Patch and restart behavior is explicit: UI state updates are persisted through card patching, and recomputation/retrigger is a deliberate step, not an implicit side effect.
  • This example is a reference surface for decisions and contracts, not a canonical list of all features. Agents should infer implementation details directly from files in this folder.

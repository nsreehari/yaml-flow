yaml-flow

Two workflow engines in one package. Pick the model that fits your problem.

Install: npm install yaml-flow

What Is yaml-flow?

yaml-flow is not only a low-level npm library. It exposes three practical consumption surfaces:

Library APIs

Embed workflow logic directly in your application:

  • yaml-flow/step-machine — sequential step execution with explicit control flow
  • yaml-flow/continuous-event-graph — Reactive DAG where tasks auto-trigger when their dependencies arrive. External events and token injections flow through a dynamic task topology. Supports runtime node mutations, cascading computation, and self-driving orchestration without explicit imperative coordination.
  • yaml-flow/batch — convenience utility for concurrent processing of item collections using Step Machine, Live Event Graph, or graph-of-graphs execution patterns
  • yaml-flow/inference — optional LLM layer for automatic task completion detection (complements continuous-event-graph)

Live-Cards / Boards Package

Reusable higher-level abstraction built using yaml-flow engines, schemas, CLI orchestration, and browser runtime pieces. It includes:

  • board-live-cards-cli
  • browser runtime assets
  • yaml-flow/card-compute

Treat this as a package layer, not just a demo. It bundles the pieces needed to build and operate LiveCards / Boards.

Step-Machine CLI

Batteries-included operational runner for YAML step-machine flows with inline handlers, CLI handlers, JSONata transforms, and store-backed run control.

  • step-machine-cli
  • --store file --store-dir ... for durable runs
  • --pause, --resume, and --status for external run control

This is the standalone operational entrypoint for step-machine workflows, especially when you want reproducible YAML-driven execution with restartable file-backed state and operator-friendly pause/resume/status controls.

Which Mode Do I Need?

yaml-flow ships two execution models solving fundamentally different problems:

Step Machine Live Event Graph
Mental model Flowchart — each step decides what runs next Dependency graph — tasks self-select when inputs are ready
Core API flow.run(input) applyEvent(live, event), schedule(live), inspect(live), addNode(live, ...)
Config shape steps: with transitions: tasks: with requires: / provides:
Who decides next? The sender (current step's result picks the transition) The receivers (tasks become eligible when their tokens appear)
Import yaml-flow/step-machine yaml-flow/continuous-event-graph

Use Step Machine when…

  • Your workflow is a known sequence of steps with conditional branching
  • You need pause/resume built in
  • The work is conversational: receive input → process → decide next step → repeat
  • You want to hand the library a YAML file and handler functions and call .run()

Examples: form wizards, approval chains, AI chat loops, order processing pipelines, ETL with linear stages.

Use Live Event Graph when…

  • Your workflow has complex dependencies — diamonds, fan-out, fan-in, optional paths
  • You don't know the execution order up front; it emerges from the data flow
  • Multiple tasks can run in parallel once their inputs are satisfied
  • You need to dynamically add or remove tasks as your system evolves
  • You need conflict resolution when two tasks compete to produce the same output
  • Your system is event-driven: external events inject tokens that unblock downstream tasks

Examples: compliance engines (GRC, audit cascades), agent tool orchestration, multi-stage data pipelines, CI/CD systems, research workflows, build systems, multi-model AI routing, eligibility engines, and perpetual monitoring topologies.

When in doubt

Start with Step Machine if your workflow diagram is a straight line with branches. Start with Live Event Graph if your workflow diagram has diamonds, parallel lanes, or needs to evolve over time. Both shift control: Step Machine puts it in the sender; Live Event Graph puts it in the data tokens.

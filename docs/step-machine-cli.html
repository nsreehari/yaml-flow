step-machine CLI Reference

Generic CLI runner for yaml-flow step-machine YAML files with inline handlers, subprocess handlers, JSONata transforms, and reusable scenario orchestration.

Invocation

step-machine-cli <step-flow.yaml> [--handlers <step-handlers.js>] [--initial-data <json>] [--store <memory|file>] [--store-dir <dir>] [--resume]
    step-machine-cli --store file --store-dir <dir> --pause
    step-machine-cli --store file --store-dir <dir> --status

Equivalent repo wrapper usage: node step-machine-cli.js ... with the same flags.

Arguments

ArgumentPurposeNotes
<step-flow.yaml>Required path to the step-machine flow file.Resolved relative to current working directory.
--handlers <step-handlers.js>Optional module exporting inline step handlers.Used when steps declare handler.inline.
--initial-data <json>Optional initial input object for the run.Must parse as a JSON object, not an array or scalar.
--store <memory|file>Optional store backend selector.Default is memory. Use file for durable run state.
--store-dir <dir>Directory for file-backed run state.Required when --store file is used.
--resumeResume the latest paused run from the configured store directory.No run ID is required. Cannot be combined with --initial-data, --pause, or --status.
--pauseRequest a pause via the store directory sentinel file.Store-level operation; used with --store file --store-dir. Cannot be combined with flow execution flags.
--statusPrint store-level run status summary for the configured store directory.Store-level operation; used with --store file --store-dir.

Step Handler Modes

ModeConfigBehavior
PassthroughNo handler blockStep input passes through unchanged, filtered by produces_data if present.
Inlinehandler: { inline: stepFn }Calls a JS function from the handlers module.
CLIhandler: { cli: "..." }Runs a subprocess with stdin JSON payload and stdout result parsing.

CLI Handler Features

  • handler_vars: top-level reusable variables. Literal strings by default; prefix with = for JSONata evaluation.
  • input-transforms: JSONata expressions evaluated against step input to produce subprocess input.
  • output-transforms: JSONata expressions evaluated against effective input plus subprocess result.
  • %%PLACEHOLDER%% templating: command placeholders resolved from transformed input.
  • result-mode: exit-code: treat zero exit as success without requiring JSON stdout.
  • produces_data: final handler output is filtered to documented data keys.

Expected Output

Successful runs print a normalized JSON result:

{
  "runId": "uuid",
  "status": "completed",
  "intent": "success",
  "finalStep": "success_state",
  "stepHistory": ["s1", "s2"],
  "data": {
    "some_key": true
  }
}

YAML Flow Structure

A step-machine YAML file defines a sequential workflow with steps, handlers, and branching logic:

id: my-flow                      # optional: flow identifier
handler_vars:                    # optional: flow-level variables for all CLI steps
  CMD: node ./handler.js         # literal string
  FLAG: "=settings.debug"        # JSONata (= prefix)
settings:
  start_step: load_config        # required: first step to run
  max_total_steps: 100           # optional: cap on total step executions
  timeout_ms: 60000              # optional: wall-clock timeout in ms
steps:
  load_config:
    description: Load configuration
    expects_data: [env]          # optional: keys required from accumulated run data
    produces_data: [config]      # optional: filter output to only these keys
    handler:
      cli: node load-config.js
      input-transforms:          # map of key -> JSONata expression
        ENV: env
      output-transforms:
        config: data.config
    transitions:
      success: validate          # keyed by the handler result string
    failure_transitions:         # optional: per-step map of result -> step
      failure: error_state
  validate:
    handler:
      inline: validateConfig
    expects_data: [config]
    produces_data: [validated]
    transitions:
      success: success_state
    failure_transitions:
      failure: error_state
terminal_states:
  success_state:
    return_intent: success
    return_artifacts: [validated]
  error_state:
    return_intent: failure
    return_artifacts: [error]

Key fields:

  • id (optional): flow identifier.
  • handler_vars (optional, top-level): flow-wide variables available to all CLI steps. Literal strings by default; prefix with = for JSONata evaluation.
  • settings.start_step: name of the first step to execute. Required.
  • settings.max_total_steps (optional): cap on total step executions across the run.
  • settings.timeout_ms (optional): wall-clock timeout in milliseconds.
  • steps: map of step definitions; keys are step names.
  • handler: defines how the step executes (inline, CLI, or passthrough).
  • transitions: per-step map of handler result string → next step name.
  • failure_transitions (optional): per-step map of failure result → step name.
  • expects_data (optional): keys required from accumulated run data passed into this step.
  • produces_data (optional): filter step output to only these keys.
  • terminal_states: map of terminal state names. Each must have return_intent. Optional return_artifacts lists keys to include in run output.

Understanding JSONata in Step Machine

JSONata is used in transforms. Both input-transforms and output-transforms are key → JSONata expression maps — not arrays.

# input-transforms: evaluated against the step's input data (expects_data keys + handler_vars)
input-transforms:
  SYMBOL_LIST: holdings[*].symbol          # array projection
  ENV: "'production'"                      # literal string (quoted in JSONata)
  TOTAL: $sum(holdings[*].qty)             # aggregate
  LABEL: "'Batch-' & $string(batch_id)"   # string concatenation

# output-transforms: evaluated against input-transforms result + subprocess output.
# In JSON mode: data key holds the parsed subprocess JSON.
# In exit-code mode: stdout key holds the raw stdout string.
output-transforms:
  saved: data.wrote                        # from subprocess JSON result
  board_dir: BOARD_DIR                     # pass through an input-transform key
  raw_out: stdout                          # available in exit-code mode
  price: data.prices.close * (1 - fee)    # computed from subprocess result

In input-transforms, the context is the step's input data merged with resolved handler_vars. In output-transforms, the context is the effective input (input data + handler_vars + input-transforms) plus subprocess output keys (data, result, stdout, stderr).

Handler Types and Composition

Inline Handlers: JavaScript functions exported from --handlers module.

// step-handlers.js
export async function loadHoldings(input) {
  return { result: 'success', data: { portfolio: [...] } };
}

// flow.yaml
handler:
  inline: loadHoldings

CLI Handlers: External processes with optional input/output transformation.

# flow-level handler_vars (top of the YAML file, not inside a step)
handler_vars:
  CMD: node ./transform.js
  ENV: "production"              # literal string
  USER_ID: "=input_user_id"     # JSONata: resolved from step input

steps:
  my_step:
    handler:
      cli: node %%CMD%%          # %%KEY%% placeholders resolved from effective input
      result-mode: exit-code     # skip JSON parsing; use stdout/stderr directly
      input-transforms:          # map of key -> JSONata expression
        PAYLOAD: "{ holdings: holdings, env: ENV }"
      output-transforms:
        result_saved: stdout

handler_vars are declared at the top level of the flow YAML (not inside a step's handler block). They are available to all CLI steps. Prefix with = for JSONata evaluation against step input:

  • SIMPLE: "value" — literal string passed as-is.
  • COMPUTED: "=user_id" — JSONata evaluated against the step's input data.
  • JOINED: "=items[*].id ~> $join(',')" — array joined to a string.

Composition Patterns

Sequential Steps: Each step's output accumulates into run data. Steps declare expects_data to receive specific keys and transitions to route to the next step.

load_config → validate → fetch_data → compute → success_state

Multi-Key Transform: Use multiple keys in input-transforms to reshape data before the subprocess:

input-transforms:
  SYMBOL_LIST: holdings[*].symbol
  ENV: "'production'"
  BATCH_LABEL: "'Batch-' & $string(batch_id)"

Error Handling with failure_transitions: Per-step map of result name → step name. The result name matches what the handler returns as result:

transitions:
  success: next_step
failure_transitions:
  failure: error_state
  timeout: retry_step

Data Passing Between Steps: Use produces_data to explicitly declare outputs, and expects_data to declare inputs:

step_a:
  produces_data: [user_id, token]
  transitions:
    success: step_b
step_b:
  expects_data: [user_id, token]
  handler:
    inline: processWithAuth

Composition Checklist

  • Name steps clearly: Use verbs and domain terms (fetch_config, validate, compute).
  • Test transforms in isolation: JSONata expressions can be complex; test each one before integrating.
  • Use produces_data: Explicitly declare what each step outputs to avoid unintended data leakage.
  • Validate input with expects_data: CLI handlers should list data dependencies.
  • Handle errors with failure_transitions: Define fallback paths for robustness.
  • Use handler_vars for config: Environment-specific values, timeouts, API keys (via = prefix).
  • Test with --initial-data: Use --initial-data '{"key": "value"}' to inject initial state and verify the flow.

Example: Portfolio Tracker Flow

id: portfolio-tracker-step-machine
handler_vars:
  BOARDCLI_CMD: node ./board-live-cards-cli.js  # literal: available as %%BOARDCLI_CMD%%
settings:
  start_step: load_holdings
  max_total_steps: 50
  timeout_ms: 120000

steps:
  load_holdings:
    description: Load portfolio holdings via inline handler
    produces_data: [holdings, user_id]
    handler:
      inline: loadHoldings        # exported from --handlers module
    transitions:
      success: fetch_prices
    failure_transitions:
      failure: failed_state

  fetch_prices:
    description: Fetch market prices for held symbols
    expects_data: [holdings]
    produces_data: [prices]
    handler:
      cli: node fetch-market-prices.js
      input-transforms:
        SYMBOLS: holdings[*].symbol   # array of symbols from holdings
      output-transforms:
        prices: data.prices           # from subprocess JSON result
    transitions:
      success: compute_total
    failure_transitions:
      failure: failed_state

  compute_total:
    description: Compute total portfolio value
    expects_data: [holdings, prices]
    produces_data: [total_value, composition]
    handler:
      inline: calculatePortfolioValue
    transitions:
      success: success_state
    failure_transitions:
      failure: failed_state

terminal_states:
  success_state:
    return_intent: success
    return_artifacts: [total_value, composition]
  failed_state:
    return_intent: failure
    return_artifacts: [error]

step-machine CLI Reference

Generic CLI runner for yaml-flow step-machine YAML files with inline handlers, subprocess handlers, JSONata transforms, and reusable scenario orchestration.

Invocation

step-machine-cli <step-flow.yaml> [--handlers <step-handlers.js>] [--data <json>] [--store <memory|file>] [--store-dir <dir>] [--resume]
    step-machine-cli --store file --store-dir <dir> --pause
    step-machine-cli --store file --store-dir <dir> --status

Equivalent repo wrapper usage: node step-machine-cli.js ... with the same flags.

Arguments

ArgumentPurposeNotes
<step-flow.yaml>Required path to the step-machine flow file.Resolved relative to current working directory.
--handlers <step-handlers.js>Optional module exporting inline step handlers.Used when steps declare handler.inline.
--data <json>Optional initial input object for the run.Must parse as a JSON object, not an array or scalar.
--store <memory|file>Optional store backend selector.Default is memory. Use file for durable run state.
--store-dir <dir>Directory for file-backed run state.Required when --store file is used.
--resumeResume the latest paused run from the configured store directory.No run ID is required. Cannot be combined with --data, --pause, or --status.
--pauseRequest a pause via the store directory sentinel file.Store-level operation; used with --store file --store-dir. Cannot be combined with flow execution flags.
--statusPrint store-level run status summary for the configured store directory.Store-level operation; used with --store file --store-dir.

Step Handler Modes

ModeConfigBehavior
PassthroughNo handler blockStep input passes through unchanged, filtered by produces_data if present.
Inlinehandler: { inline: stepFn }Calls a JS function from the handlers module.
CLIhandler: { cli: "..." }Runs a subprocess with stdin JSON payload and stdout result parsing.

CLI Handler Features

  • handler_vars: top-level reusable variables. Literal strings by default; prefix with = for JSONata evaluation.
  • input-transforms: JSONata expressions evaluated against step input to produce subprocess input.
  • output-transforms: JSONata expressions evaluated against effective input plus subprocess result.
  • %%PLACEHOLDER%% templating: command placeholders resolved from transformed input.
  • result-mode: exit-code: treat zero exit as success without requiring JSON stdout.
  • produces_data: final handler output is filtered to documented data keys.

Expected Output

Successful runs print a normalized JSON result:

{
  "runId": "uuid",
  "status": "completed",
  "intent": "success",
  "finalStep": "success_state",
  "stepHistory": ["s1", "s2"],
  "data": {
    "some_key": true
  }
}

YAML Flow Structure

A step-machine YAML file defines a sequential workflow with steps, handlers, and branching logic:

name: portfolio-tracker
initial_state: idle
steps:
  holdings:
    handler:
      inline: loadHoldings
    transition_on_complete: fetch-prices
  fetch-prices:
    handler:
      cli: node fetch.js
      input-transforms: ["$rename(state, { holdings: '$x' })"]
      output-transforms: ["prices: sources.prices"]
    requires:
      - holdings
    transition_on_complete: compute-value
  compute-value:
    handler:
      inline: calculateTotal
    produces_data: [total_value, currency]
    transition_on_complete: success_state
  success_state:
    handler:
    produces_data: [total_value, currency]
failure_transitions:
  - from: "*"
    to: failure_state
    if: "error"
  failure_state:
    handler:

Key fields:

  • steps: map of step definitions; keys are step names.
  • handler: defines how the step executes (inline, CLI, or passthrough).
  • transition_on_complete: next step name after success.
  • requires (CLI handlers): array of data keys that must be in input.
  • input-transforms: JSONata expressions to shape input before passing to handler.
  • output-transforms: JSONata expressions to shape handler result.
  • produces_data: filter final output to only these keys.
  • failure_transitions: routes to error states on failure.

Understanding JSONata in Step Machine

JSONata is used in transformations to reshape and derive data at each step:

// Rename and restructure
"$rename(state, { old_key: 'new_key' })"

// Pick specific fields
"{ price: state.price, qty: state.qty }"

// Array mapping
"state.items.{ id, quantity * 2 as double_qty }"

// Conditional transforms
"state.discount_rate > 0 ? { final_price: state.price * (1 - state.discount_rate) } : { final_price: state.price }"

// Merge state with handler result
"state & { result_field: sources.handler_result }"

// Combine multiple sources
"{ summary: sources.prices, transactions: requires.history }"

In input-transforms, the context is the step input. In output-transforms, you have access to both.

Handler Types and Composition

Inline Handlers: JavaScript functions exported from --handlers module.

// step-handlers.js
export async function loadHoldings(input) {
  return { portfolio: [...] };
}

// flow.yaml
handler:
  inline: loadHoldings

CLI Handlers: External processes with optional input/output transformation.

handler:
  cli: node transform.js
  handler_vars:
    ENV: "production"
    DEBUG: "=state.debug"  // JSONata evaluation (= prefix)
  input-transforms:
    - "{ holdings: state.holdings, env: handler_vars.ENV }"
  output-transforms:
    - "{ total: sources.result, timestamp: $now() }"
  result-mode: exit-code

handler_vars: Reusable variables injected into CLI command environment. Prefix with = for JSONata evaluation:

  • SIMPLE: "value" — literal string.
  • COMPUTED: "=state.user_id" — evaluated JSONata.
  • ARRAY: "=requires.items.$map($string(id))" — array transform to string.

Composition Patterns

Sequential Steps: Each step's output passes to the next via transition_on_complete.

load_config → validate → fetch_data → compute → save

Multi-Transform Pipeline: Chain multiple JSONata expressions to progressively reshape data:

input-transforms:
  - "requires.raw_data"
  - "{ normalized: $uppercase($) }"
  - "{ final: { value: $, timestamp: $now() } }"

Error Handling with failure_transitions: Catch failures and route to error steps:

failure_transitions:
  - from: fetch_data
    to: retry_fetch
  - from: "*"
    to: log_error
    if: "error"

Data Passing Between Steps: Use produces_data to explicitly declare outputs, and requires to document inputs:

step_a:
  produces_data: [user_id, token]
step_b:
  requires: [user_id, token]
  handler:
    inline: processWithAuth

Composition Checklist

  • Name steps clearly: Use verbs and domain terms (fetch_config, validate, compute).
  • Test transforms in isolation: JSONata expressions can be complex; test each one before integrating.
  • Use produces_data: Explicitly declare what each step outputs to avoid unintended data leakage.
  • Validate input with requires: CLI handlers should list data dependencies.
  • Handle errors with failure_transitions: Define fallback paths for robustness.
  • Use handler_vars for config: Environment-specific values, timeouts, API keys (via = prefix).
  • Test with --data: Use --data '{"key": "value"}' to inject initial state and verify the flow.

Example: Portfolio Tracker Flow

name: portfolio-tracker
initial_state: idle

steps:
  load-holdings:
    handler:
      inline: loadHoldings
    produces_data: [holdings, user_id]
    transition_on_complete: fetch-prices

  fetch-prices:
    handler:
      cli: node fetch-market-prices.js
      handler_vars:
        SYMBOLS: "=requires.holdings[*].symbol"
        TIMEOUT: "30"
      input-transforms: ["{ symbols: handler_vars.SYMBOLS }"]
      output-transforms: ["prices: sources.market_data"]
    requires: [holdings]
    produces_data: [prices]
    transition_on_complete: compute-total

  compute-total:
    handler:
      inline: calculatePortfolioValue
      input-transforms: ["{ holdings: requires.holdings, prices: requires.prices }"]
    requires: [holdings, prices]
    produces_data: [total_value, composition]
    transition_on_complete: success_state

  success_state:
    handler:

failure_transitions:
  - from: "*"
    to: failure_state
    if: error

failure_state:
  handler:

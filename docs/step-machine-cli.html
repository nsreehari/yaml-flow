step-machine CLI Reference

Generic CLI runner for yaml-flow step-machine YAML files with inline handlers, subprocess handlers, JSONata transforms, and reusable scenario orchestration.

Invocation

step-machine-cli <step-flow.yaml> [--handlers <step-handlers.js>] [--data <json>]

Equivalent repo wrapper usage: node step-machine-cli.js <step-flow.yaml> [--handlers <step-handlers.js>] [--data <json>]

Arguments

ArgumentPurposeNotes
<step-flow.yaml>Required path to the step-machine flow file.Resolved relative to current working directory.
--handlers <step-handlers.js>Optional module exporting inline step handlers.Used when steps declare handler.inline.
--data <json>Optional initial input object for the run.Must parse as a JSON object, not an array or scalar.

Step Handler Modes

ModeConfigBehavior
PassthroughNo handler blockStep input passes through unchanged, filtered by produces_data if present.
Inlinehandler: { inline: stepFn }Calls a JS function from the handlers module.
CLIhandler: { cli: "..." }Runs a subprocess with stdin JSON payload and stdout result parsing.

CLI Handler Features

  • handler_vars: top-level reusable variables. Literal strings by default; prefix with = for JSONata evaluation.
  • input-transforms: JSONata expressions evaluated against step input to produce subprocess input.
  • output-transforms: JSONata expressions evaluated against effective input plus subprocess result.
  • %%PLACEHOLDER%% templating: command placeholders resolved from transformed input.
  • result-mode: exit-code: treat zero exit as success without requiring JSON stdout.
  • produces_data: final handler output is filtered to documented data keys.

Expected Output

Successful runs print a normalized JSON result:

{
  "runId": "uuid",
  "status": "completed",
  "intent": "success",
  "finalStep": "success_state",
  "stepHistory": ["s1", "s2"],
  "data": {
    "some_key": true
  }
}

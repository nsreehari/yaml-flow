board-live-cards CLI Reference

Canonical command reference for board runtime initialization, card lifecycle operations, source execution, and status contracts.

Invocation

board-live-cards-cli <command> [options]

Equivalent repo wrapper usage: node board-live-cards-cli.js <command>

LiveCard JSON Structure

Each card is a JSON file that declares dependencies, data sources, and computed outputs:

{
  "id": "portfolio-value",
  "meta": { "title": "Portfolio Valuation" },
  "requires": ["holdings", "prices"],
  "provides": [{ "bindTo": "total_value", "src": "computed_values.value" }],
  "state": { "initial_data": 100 },
  "sources": [
    { "cli": "node fetch-market.js", "bindTo": "market", "outputFile": "market.json" }
  ],
  "compute": [
    { "bindTo": "value", "expr": "$sum(requires.holdings.(qty * $lookup(requires.prices, symbol).price))" }
  ],
  "view": {
    "elements": [
      { "kind": "metric", "label": "Total Value", "data": { "bind": "computed_values.value" } }
    ]
  }
}

Key fields:

  • id: unique card identifier (becomes a task in the board graph).
  • requires: array of token names that must be available before this card runs.
  • provides: expressions that bind card outputs to tokens for downstream cards; uses JSONata src.
  • state: initial data object available during compute as state.*.
  • sources: task executor-owned data-fetch descriptors; opaque to this CLI tool — interpreted and dispatched by the consuming system's task executor. Results are made available as sources.*. See Source Execution Notes below.
  • compute: JSONata expressions that derive new values from state, requires, sources, and earlier compute results.

Understanding JSONata in LiveCards

JSONata is a lightweight query and transformation language for JSON. In card compute steps and provides[].src, you write JSONata expressions that transform data:

// Access state properties
"state.price"

// Array operations
"$sum(state.quantities)"
"$average(requires.history.values)"

// Conditional logic
"state.quantity > 100 ? state.quantity * 0.9 : state.quantity"

// Lookup and grouping
"$lookup(requires.prices, symbol).price"
"requires.transactions{category: [$sum(amount)]}"

// Multi-step chains (access previous compute results)
"computed_values.subtotal * computed_values.discount_factor"

Learn more at jsonata.org

Card Runtime Lifecycle

When a card becomes eligible (all requires tokens satisfied), the runtime executes the following sequence:

1. Card becomes eligible — all requires tokens satisfied by upstream task-completed events
    2. Read card.state from disk
    3. Inject requires data (read-only, from upstream task-completed.data — immutable during this run)
    4. Run compute array top-to-bottom → export a per-run runtime snapshot with computed_values, card_data, and sources_data for file-based consumers
    5. Check: all sources[].bindTo keys present in state?
   YES → Build provides data (pluck paths from state, requires, computed_values per provides[].src)
         Persist each provided token to runtime-out/data-objects/<tokenName>
         Emit task-completed with provides data → unblocks downstream cards
         Spawn any non-gating sources (optionalForCompletionGating: true) in background
   NO  → Spawn all outstanding gating sources
         Emit task-initiated (card stays in-progress, downstream stays blocked)
6. Source delivers → writes state[bindTo] on disk → emits source-data-fetched → card re-fires from step 1
7. Non-gating source delivers → same write/re-fire → card re-completes with richer data

The two data namespaces are intentionally separate: requires.* is read-only, injected from upstream tokens and immutable during the card's execution. state.* is the card's persistent local store on disk, seeded by the card author and mutated only by source delivery at runtime. This separation prevents self-referential compute loops.

Events

Task-Executor Protocol

When .task-executor is registered in the board directory, the board runtime delegates source execution to that external script. The executor must implement the run-source-fetch subcommand contract:

<executor-command> run-source-fetch --in <sourceDefFile> --out <resultFile> --err <errorFile>

All paths are file paths (not JSON strings). The executor receives a JSON source definition on disk and must produce output files atomically:

ArgumentDescription
--in <file>Path to source definition JSON file. Contains { cli, bindTo, cwd, timeout, ... } properties.
--out <file>Path where executor must write the execution result (typically stdout from running source.cli).
--err <file>Optional path where executor writes error messages if execution fails.

Exit Codes & Behavior:

  • Exit 0 — Success. The runtime reads --out file and invokes source-data-fetched callback.
  • Exit non-zero — Failure. The runtime reads --err file and invokes source-data-fetch-failure callback.

Executor Registration: Place the executor command (as a single-line string) in a .task-executor file at the board root, or pass --task-executor <command> during init:

board-live-cards-cli init ./my-board --task-executor "node ./my-executor.js"
echo "node ./my-executor.js" > ./my-board/.task-executor

Example Implementation: See examples/browser/boards/portfolio-tracker/portfolio-tracker-task-executor.cjs for a complete working executor that runs CLI commands synchronously and captures output.

Refresh Strategy

This is part of the board runtime scheduling model, not the browser renderer. It controls when a completed card becomes eligible to run again.

StrategyBehavior
data-changedDefault. Re-run when upstream output data changes.
epoch-changedRe-run when upstream execution count advances, even if data is identical.
time-basedRe-run after refreshInterval seconds.
manualNever auto-rerun. Only external retrigger or operator action can restart it.
onceRun one time and never become eligible again.
 
// Per-card config
{ "refreshStrategy": "time-based", "refreshInterval": 60, "maxExecutions": 10 }

// Board default
{ "settings": { "refreshStrategy": "epoch-changed" } }

In this repo, the scheduling logic for these strategies lives in the event-graph / continuous-event-graph runtime, and board-live-cards-cli uses that runtime to decide card re-eligibility.

Infinite Loop Prevention

The runtime is designed to be loop-safe by construction:

  1. Compute runs exactly once per invocation — no iterative convergence, no polling. The array runs top-to-bottom and stops.
  2. A card re-fires only on external events — upstream requires change, source-data-fetched delivery, or manual retrigger. Not on its own compute output.
  3. Compute output is an exported snapshot, not durable graph state — all computed_values are rewritten to the per-card runtime artifact on each run so file-based consumers can read the latest computed result set.
  4. Namespace separation prevents self-reference — state.x in a compute expr reads the persisted seed value, never the current invocation's compute output. Writing to computed_values.x does not overwrite state.x.

Disk Layout

The board runtime maintains board state under the board directory and render/runtime artifacts under the configured runtime-out directory:

  • cards/<id>.json — the card definition files (authored by the user). Contains id, requires, provides, state, sources, compute, meta, view. Sources are the only runtime writers to a card's state; the card definition file itself is not modified by the runtime.
  • board-graph.json — runtime task graph. Stores task statuses (not-started, in-progress, completed, failed) and the data emitted by each card's task-completed event (the provides tokens). This is what downstream cards read as their requires.* namespace.
  • board-journal.jsonl — append-only event log. Every event (task-completed, source-data-fetched, task-failed, etc.) is appended here. Used for audit, debugging, and replay.
  • .runtime-out — pointer file storing the runtime artifact directory path. Defaults to <boardDir>/runtime-out when first resolved.
  • runtime-out/data-objects/<tokenName> — persisted JSON payload for each provided token.
  • runtime-out/cards/<cardId>.computed.json — exported per-card runtime snapshot used by server and UI layers; rewritten each time the card computes.

Source output files are written to <boardDir>/<cardId>/<bindTo>.json (or .txt) by the built-in local executor and are the mechanism by which source delivery is communicated back to the card runtime.

Sequential Data Flow: Cards depend on tokens provided by upstream cards. The board scheduler ensures dependencies are satisfied.

holdings (card) —[provides: holdings]—> portfolio-value (requires: holdings)

Compute Chaining: Later compute steps in a single card can reference results from earlier steps via computed_values.*:

"compute": [
  { "bindTo": "subtotal", "expr": "$sum(state.items.price)" },
  { "bindTo": "tax", "expr": "computed_values.subtotal * 0.1" },
  { "bindTo": "total", "expr": "computed_values.subtotal + computed_values.tax" }
]

Source Output Binding: CLI sources write JSON or text to outputFile. Use provides to expose the result as a token:

"sources": [{ "cli": "node fetch-prices.js", "bindTo": "prices", "outputFile": "prices.json" }],
"provides": [{ "bindTo": "prices", "src": "sources.prices" }]

Card Composition Checklist

  • Declare requires: Name all token dependencies explicitly.
  • Use provides for outputs: Even leaf cards should declare what they provide to downstream consumers.
  • Keep compute expressions simple: Test each JSONata expression in isolation if possible.
  • Required vs enrichment sources: By default, all sources gate card completion — the card stays in-progress and downstream cards remain blocked until every source is delivered. Set optionalForCompletionGating: true on sources that are view enrichment only: they run in the background and their data enhances the card's display (e.g. an LLM sentiment indicator, a secondary metric, a sparkline), but the card completes and unblocks downstream cards without waiting for them.
  • Timeout sources: Set timeout (ms) for long-running CLIs.
  • Test with status --json: Verify card state and topology before running the full board.

Source Execution Notes

The sources array on a card is opaque to this CLI tool. It is passed through as-is to whatever task executor the consuming system provides. Systems integrating this tool are expected to bring their own task executor that understands the sources shape and handles dispatch, retries, and output delivery.

The notes below apply only to the built-in local executor, which is used as a fallback when no external task executor is configured (e.g. in standalone or development use):

  • source.cli is parsed with quote-aware argument splitting and executed without shell wrapping.
  • Stdout is captured and written to source output artifacts (filepath: boardDir/cardId/bindTo.json or .txt).
  • Timeout defaults to 120s unless overridden by source.timeout.

Completion gating (regardless of executor): Sources gate completion by default (including when optionalForCompletionGating is omitted or set to false). Set optionalForCompletionGating: true for view-enrichment-only sources: they run in the background and never block task-completed or downstream cards.

{
  "id": "holdings",
  "state": { "portfolio": [{ "symbol": "AAPL", "qty": 10 }] },

}

// cards/price-fetch.json
{
  "id": "price-fetch",
  "requires": ["holdings"],
  "sources": [{ "cli": "node fetch-prices.js", "bindTo": "prices", "outputFile": "prices.json" }],
  "provides": [{ "bindTo": "prices", "src": "sources.prices" }]
}

{
  "id": "portfolio-value",
  "requires": ["holdings", "prices"],
  "compute": [
    { "bindTo": "total", "expr": "$sum(requires.holdings.(qty * requires.prices[symbol].price))" }
  ],
  "provides": [{ "bindTo": "portfolio_value", "src": "computed_values.total" }]

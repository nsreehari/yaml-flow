Browser Runtime: LiveCards & Boards

The files in browser/ are a public browser-facing runtime surface that complements yaml-flow library APIs and CLIs. They are plain scripts — no bundler required — designed to be dropped into any HTML page.

Core Concepts

LiveCard (Node)

A LiveCard is a JSON entity (also called a node) that declares its data shape and visual presentation declaratively. Behavior is determined by which sections are present — there is no explicit type field:

  • id — unique identifier within a board
  • card_data — authored seed data; the card's persistent data namespace, mutated by source_defs at runtime
  • requires — upstream node IDs whose output tokens this card depends on (read-only at runtime)
  • provides — output tokens exposed to downstream cards
  • source_defs — opaque data-fetch descriptors; dispatched by the task executor, not the browser engine. Only bindTo and outputFile matter to the engine; all other fields are passed verbatim.
  • compute — ordered JSONata expressions derived from card_data.*, requires.*, and earlier compute results; writes to ephemeral computed_values.* (never persisted)
  • view — declarative rendering spec; if present the node renders as a visual card; if absent but source_defs are present, it renders as a source pill — a compact inline indicator showing the source name and fetch status, without a full card UI

Board

A Board is a collection of LiveCard nodes arranged in a 2D layout. It wraps an engine and a DOM container and manages spatial layout, node addition/removal, and two display modes:

  • board mode — standard grid/list layout of cards
  • canvas mode — free-form positioning with dependency edges visible

Boards are created via LiveCard.Board(engine, containerEl, { nodes, positions?, mode, canvas }).

Engine

The engine (LiveCard.init({...})) is the core runtime object. It manages rendering individual nodes into DOM elements, tracks subscriptions between dependent nodes, runs compute expressions when upstream data changes, and dispatches notifications. Multiple boards can share one engine.

What Is Shipped

  • browser/live-cards.js — the Board/Canvas engine and card rendering runtime. Exposes LiveCard.init(cfg) to create an engine and LiveCard.Board(engine, el, opts) to mount a board. Depends on Bootstrap 5 for layout; uses card-compute.js for compute expressions; optionally uses Chart.js for chart elements.
  • browser/card-compute.js — the JSONata-powered compute engine for LiveCard nodes. Isomorphic: the same logic is also available as the yaml-flow/card-compute npm import for Node.js / bundler use. Requires JSONata to be loaded first (<script src="https://cdn.jsdelivr.net/npm/jsonata/jsonata.min.js">). API: CardCompute.run(node), CardCompute.eval(expr, node), CardCompute.resolve(node, path), CardCompute.validate(node).
  • browser/live-cards.schema.json — the live-cards JSON Schema contract, mirrored here so browser consumers can validate card definitions without a server-side dependency.

Runtime Dependencies

  • Bootstrap 5 — required by live-cards.js for layout, modals, and component classes
  • JSONata — required by card-compute.js; must be loaded before it
  • Chart.js — optional; if present enables chart elements in card views; falls back to a table if absent
  • marked — optional; if present enables markdown rendering in card text elements
  • DOMPurify — optional; recommended alongside marked to sanitize rendered HTML

How It Fits

The browser runtime is not a separate product. It is the browser-facing layer of the live-cards / boards package, which spans:

  • Server side: board-live-cards-cli orchestrates task lifecycle, manages board state on disk, and communicates with the browser page via SSE or file polling.
  • Compute: yaml-flow/card-compute (Node) / browser/card-compute.js (browser) — same isomorphic engine for JSONata expressions in both environments.
  • Browser side: browser/live-cards.js renders cards and boards, re-runs compute when upstream data changes, and delegates all data-fetch and state-persistence side-effects back to the host.

Note: source_defs[] on nodes are opaque to the browser engine — only bindTo and outputFile are consumed by the engine. All other fields are owned by the task executor and passed through verbatim.

Typical Usage Pattern

<!-- 1. Load dependencies -->
<script src="https://cdn.jsdelivr.net/npm/jsonata/jsonata.min.js"></script>
<link rel="stylesheet" href="bootstrap.min.css" />
<script src="bootstrap.bundle.min.js"></script>

<!-- 2. Load browser runtime -->
<script src="browser/card-compute.js"></script>
<script src="browser/live-cards.js"></script>

<script>
// 3. Build a lookup map from your card definitions
const nodesMap = Object.fromEntries(cardDefinitions.map(n => [n.id, n]));

// 4. Create engine — host supplies resolve + side-effect callbacks
const engine = LiveCard.init({
  resolve:      id => nodesMap[id],          // return node by id
  onPatch:      (id, patch) => { /* persist source_defs/compute results */ },
  onPatchState: (id, patch) => { /* persist card_data mutations */ },
  onRefresh:    () => { /* re-fetch board state */ },
});

// 5. Mount a board
const board = LiveCard.Board(engine, document.getElementById('board'), {
  nodes: cardDefinitions,
  mode: 'board',
});
</script>

Server-side CLI (board-live-cards-cli) or your own task executor handles all data fetching, source dispatch, and disk card-data management. The browser engine is a pure render + compute layer.

Built-in View Element Kinds

Each card's view.elements[] array defines what renders on that card. Each element has a required kind field and an optional label. Data-display elements take a data.bind path pointing into the card's data namespaces — card_data.*, requires.*, computed_values.*, or fetched_sources.*. Interactive elements like chat, file-upload, and actions do not require a data binding. The 16 built-in kinds are:

Data Display

  • table — render an array of objects as a table
  • list — render an array as a bulleted list (falls back to table if no data)
  • chart — render array data as a line/bar/pie chart (requires Chart.js; falls back to table if absent)
  • metric — large, bold number with optional label and detail text
  • badge — colored badge/pill with status or category

Text & Markdown

  • text — plain text (inline or wrapping)
  • markdown — markdown-rendered text (requires marked; falls back to plain text if absent)
  • narrative — block of markdown for longer-form narrative content

User Input & Interaction

  • form — declarative form with text, checkbox, select, and textarea fields; writes back to card data via optional writeTo
  • filter — key-value input fields bound to card data; typically used for filtering or search parameters
  • notes — textarea with markdown preview side-by-side; persistent in card data
  • todo — checklist of todo items; each can be marked complete

Status & Alerts

  • alert — colored alert box (green, amber, red) with status dot and text

Special Elements

  • chat — built-in chat interface (messages + input); supports disabled state via card_data.features.chat.disabled
  • file-upload — drag-drop or click-to-upload zone; typically paired with chat; supports disabled state via card_data.features.files.disabled
  • actions — buttons or action controls (e.g., confirm, discard, retry); host-wired callbacks

Custom Elements

Register custom rendering functions via engine.registerRenderer(name, fn). The function receives (data, el, elemDef, node) and renders into the DOM element.

// Register a custom "timeline" renderer
engine.registerRenderer('timeline', (data, el, elemDef, node) => {
  // data is the resolved JSONata expression
  // el is the container div
  // elemDef is the view element definition { kind, label, data, ... }
  // node is the full LiveCard node
  el.innerHTML = '<div class="timeline">...</div>';
});

Then use it in card views:

"view": {
  "elements": [
    { "kind": "timeline", "label": "Activity Feed", "data": { "bind": "card_data.events" } }
  ]
}

Disabling Chat and Files on a Card

Set card_data.features.chat.disabled or card_data.features.files.disabled to true to make a card read-only without hiding its history. The engine reads these flags each time the modal is opened, so the host can toggle them at runtime by patching card_data.

FieldEffect when trueStill visible
card_data.features.chat.disabled Chat input, attach button, and send button are disabled; placeholder text changes to “Chat is disabled for this card.” Full message history, close button
card_data.features.files.disabled Select files button, Upload button, and file input are disabled; dropzone gets lc-disabled styling (dimmed, no pointer) Previously uploaded file list and all download links

Minimal example — an inactive ingest card:

{
  "id": "ingest-batch-001",
  "card": {
    "meta": { "title": "Invoice Batch 2025-04-15" },
    "view": {
      "elements": [
        { "kind": "text",        "label": "Status",    "data": { "bind": "card_data.status" } },
        { "kind": "chat",        "label": "Chat" },
        { "kind": "file-upload", "label": "Files" },
        { "kind": "actions",     "label": "Actions",
          "data": { "actions": [ { "label": "Confirm", "actionType": "confirm" } ] } }
      ]
    }
  },
  "card_data": {
    "status": "processed",
    "features": {
      "chat":  { "disabled": true },
      "files": { "disabled": true }
    }
  }
}

The host activates or deactivates a card by patching card_data.features.* via onPatchState. No schema changes to view are needed — the same card definition covers both active and inactive states.
